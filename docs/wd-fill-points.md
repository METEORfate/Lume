## 0 需求分析和调研

### 项目背景与调研

- Web 服务是现代互联网应用的基础，企业官网、博客、后台管理系统和前端项目都需要 Web 服务器提供 HTTP 静态资源访问能力。
- 常见 Web 服务器包括 Apache、Nginx、LiteSpeed 等。
- Apache 功能丰富但整体较重；Nginx 采用事件驱动模型，高并发能力强；LiteSpeed 性能较好但偏商业化。
- 成熟服务器源码规模大、模块复杂，不适合初学者直接理解底层网络模型。
- 本项目通过自行实现轻量级静态 Web 服务器，帮助理解 Linux 网络编程、HTTP 协议、epoll I/O 多路复用、非阻塞连接处理和模块化 C 工程设计。

### 项目目标

- 使用 C 语言在 Linux 平台实现轻量级静态 Web 服务器。
- 支持 HTTP GET 请求，能够返回 HTML、CSS、JS、图片等静态资源。
- 使用 epoll 实现单线程 Reactor 事件驱动模型。
- 支持配置文件读取，避免端口和根目录硬编码。
- 支持基础 HTTP 状态码：`200`、`400`、`404`、`501`。
- 实现 URI 到本地文件路径的安全映射，防止目录遍历攻击。
- 保持项目结构清晰、模块解耦，方便后续性能优化。

### 用户角色分析

- 普通浏览器用户：通过浏览器访问静态网页。
- 网站维护者：把自己的静态网站文件放入 `public/` 或配置指定目录。
- 开发/学习者：通过阅读源码学习 Linux C 网络服务器实现。
- 测试人员：通过 `curl`、冒烟测试、压测脚本验证服务器行为。

### 痛点分析与竞品对标

- 学习痛点：Nginx/Apache 源码复杂，学习门槛高。
- 部署痛点：小型静态网站不一定需要完整大型服务器。
- 工程痛点：手写 socket 服务容易出现阻塞、资源泄漏、路径不安全等问题。
- 对标 Nginx：本项目功能远少于 Nginx，但保留事件驱动、高并发、静态资源服务等核心思想。
- 对标 Python 简易 HTTP 服务：本项目更贴近 Linux 系统编程底层，能体现 epoll、非阻塞 I/O 和 C 工程管理。

### 可行性分析

- 技术可行：Linux 提供 socket、epoll、fcntl、stat、open/read/write 等完整 API。
- 功能可行：静态 Web 服务范围明确，第一阶段无需动态后端和数据库。
- 工程可行：项目采用 CMake 管理，模块边界清晰，便于编译、测试和维护。
- 测试可行：可使用 `curl`、单元测试、冒烟测试和压测脚本进行验证。

### 1.2 定义

- HTTP：超文本传输协议，浏览器与服务器通信的基础协议。
- GET：HTTP 请求方法，用于请求服务器资源。
- URI：统一资源标识符，本项目中主要表示浏览器请求的文件路径。
- MIME Type：资源类型标识，例如 `text/html`、`text/css`、`image/png`。
- epoll：Linux I/O 多路复用机制，用于高效监听多个文件描述符事件。
- Reactor：事件驱动设计模式，主循环等待事件并分发给对应处理模块。
- 非阻塞 I/O：调用 `read/write/accept` 时如果暂时不能完成，立即返回而不是阻塞线程。
- 短连接：一次请求响应完成后主动关闭 TCP 连接。
- ROOT_DIR：配置文件中的静态文件根目录。
- 目录遍历攻击：通过 `../` 等路径访问 Web 根目录外部文件的攻击方式。

### 2.2 主要功能和性能

#### 主要功能

- 配置文件加载：读取端口、静态根目录、请求头限制、连接数限制和超时时间。
- 网络监听：创建 TCP socket，绑定端口并监听客户端连接。
- epoll 事件循环：使用 Linux epoll 管理监听 fd 和客户端 fd。
- HTTP 请求解析：解析请求方法、URI 和 HTTP 版本。
- 静态文件服务：返回 HTML、CSS、JS、图片、JSON、文本等文件。
- MIME 识别：根据扩展名设置 `Content-Type`。
- 路径安全：拒绝 `..`、非法 URL 编码、空字节、控制字符和越界路径。
- 状态码处理：支持 `200 OK`、`400 Bad Request`、`404 Not Found`、`501 Not Implemented`。
- 连接状态管理：按读请求、处理请求、写响应头、写文件、关闭连接分阶段处理。
- 基础测试：提供单元测试、冒烟测试和压测脚本。

#### 性能特点

- 使用 epoll 单线程 Reactor 模型，避免一个连接一个线程的高资源消耗。
- socket 设置为非阻塞，提高并发连接处理能力。
- accept/read/write 均按非阻塞方式处理。
- 文件内容分块读取和发送，避免一次性读入大文件。
- 当前性能优化以稳定性为主，后续可加入 `sendfile`、缓冲区复用和边缘触发 epoll。

### 试运行结果

- 首页、CSS、JS 等静态资源可正常访问。
- 不存在文件返回 `404 Not Found`。
- 非 GET 请求返回 `501 Not Implemented`。
- 目录遍历请求返回 `400 Bad Request`。
- 超长请求头返回 `400 Bad Request`。
- 大文件下载能够完整返回。

### 试运行中发现的问题和处理

- 问题一：路径安全风险。处理方式：对 URI 解码后检查 `..`、控制字符、空字节，并使用 `realpath` 判断最终路径是否仍在根目录内。
- 问题二：慢连接和空闲连接可能长期占用 fd。处理方式：增加 `REQUEST_TIMEOUT`，事件循环定期清理空闲连接。
- 问题三：请求头过大可能占用内存。处理方式：增加 `MAX_REQUEST_BYTES` 限制。
- 问题四：并发连接数过多可能耗尽资源。处理方式：增加 `MAX_CONNECTIONS` 限制。
- 问题五：大文件发送可能出现 partial write。处理方式：连接模块保存发送偏移，分块发送直到完成。

- 掌握 socket 创建、绑定、监听和接受连接流程。
- 理解阻塞 I/O 与非阻塞 I/O 的区别。

- 掌握 `epoll_create1`、`epoll_ctl`、`epoll_wait` 基本用法。
- 理解事件驱动模型中“等待事件、分发事件、处理事件”的流程。
- 理解监听 fd 和客户端 fd 在 epoll 中的不同处理方式。
- 理解水平触发模式更容易实现正确，适合作为初始版本。

### HTTP 协议基础

- 掌握 HTTP 请求行结构：`METHOD URI VERSION`。
- 掌握常见响应状态码的含义。
- 理解响应头 `Content-Length`、`Content-Type`、`Connection` 的作用。
- 理解静态资源服务器如何根据 URI 返回本地文件。

### 文件系统与安全

- 掌握 `stat` 获取文件元信息。
- 掌握 `open/read/write/close` 文件 I/O。
- 理解目录遍历攻击的风险。
- 掌握使用 `realpath` 检查路径是否逃逸根目录。

### C 工程化

- 掌握 CMake 管理多源文件项目。
- 掌握头文件和源文件分离。
- 掌握模块化设计和接口封装。

### 不足

- HTTP 协议支持仍较基础，未支持 keep-alive、缓存协商、Range 请求等功能。
- 性能优化还不够充分，暂未使用 `sendfile`。
- 日志系统较简单，暂未实现完整访问日志和统计指标。
- 压测数据还不够系统，缺少不同并发数下的性能对比。
- 单页应用 history 路由 fallback 暂未支持。

### 未来改进方向

- 引入 `sendfile`，减少文件发送过程中的数据拷贝。
- 增加访问日志，记录请求方法、URI、状态码、响应字节数和耗时。
- 增加更多配置项，如 `BACKLOG`、`MAX_EVENTS`、日志级别等。
- 增加 SPA fallback 配置，支持前端单页应用部署。
- 支持 HTTP keep-alive，提高短时间多资源访问效率。
- 增加压测报告，记录 QPS、平均延迟、P95/P99 延迟、错误率等指标。
- 持续完善异常测试，包括慢请求、断连、大量并发、非法编码等场景。

### 工程实践经验

- 凡是系统调用都有失败可能，必须检查返回值。
- 凡是涉及指针和内存分配，都要处理空指针和释放问题。
- 凡是涉及文件路径，都要防止目录遍历和越界访问。
- 非阻塞 I/O 必须正确处理 `EAGAIN`。
- 大文件发送不能假设一次 `write` 就能完成，必须维护发送偏移。
- 项目早期就要重视模块划分，否则后续维护成本会明显增加。
- 测试脚本和文档不是附属内容，而是保证项目可运行、可复现、可交付的重要组成部分。
