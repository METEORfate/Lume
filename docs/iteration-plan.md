# Lume 静态 Web 服务器迭代开发计划

本文档记录项目各阶段的主要开发任务。整体策略是：第一阶段先实现完整可用、
业务逻辑可靠的服务器；后续阶段再逐步优化非阻塞 I/O、性能、稳定性和工程扩展能力。

## 阶段一：完整可用版

目标：完成一个可以编译、运行、访问静态文件，并具备基础安全防护的完整服务器。

主要开发任务：

- 建立完整 C 工程结构，包括 `CMakeLists.txt`、`src/`、`include/`、`tests/`、`config/`、`public/`、`scripts/`、`docs/`。
- 实现配置加载模块，支持 `PORT` 和 `ROOT_DIR`。
- 校验端口合法性，校验静态文件根目录是否存在。
- 实现服务器启动流程，包括创建 socket、设置 `SO_REUSEADDR`、绑定端口和监听连接。
- 将监听 socket 设置为非阻塞。
- 实现基础 `epoll` Reactor 事件循环。
- 支持监听 fd 事件处理，能够接受新客户端连接。
- 支持客户端 fd 事件处理，能够读取请求、生成响应、发送响应并关闭连接。
- 实现 HTTP 请求行解析，提取请求方法、URI 和 HTTP 版本。
- 支持 `GET` 请求。
- 对非 `GET` 方法返回 `501 Not Implemented`。
- 对非法请求返回 `400 Bad Request`。
- 实现标准 HTTP 响应构造。
- 支持 `200 OK`、`400 Bad Request`、`404 Not Found`、`501 Not Implemented`。
- 响应头包含 `Content-Length`、`Content-Type`、`Connection: close`、`Server`。
- 实现 URI 到本地文件路径的映射。
- 将 `/` 默认映射为 `/index.html`。
- 使用 `stat` 获取静态文件大小和类型信息。
- 使用 `open/read/write` 分块读取并发送静态文件。
- 根据扩展名识别 MIME 类型。
- 实现目录遍历防护，拒绝 `..`、非法 URL 编码、空字节、控制字符和越界路径。
- 增加基础单元测试，覆盖配置解析、HTTP 解析、MIME 映射和静态路径安全。
- 增加冒烟测试脚本，验证 200、400、404、501 和静态资源访问。
- 编写基础架构文档和新手 README。

阶段一暂不处理：

- `sendfile` 零拷贝。
- HTTP keep-alive。
- 线程池。
- `EPOLLET` 边缘触发。
- TLS。
- 动态内容。

验收标准：

- 项目可以通过 CMake 编译。
- 服务器可以通过配置文件启动。
- 浏览器或 `curl` 可以访问 HTML、CSS、JS、图片等静态资源。
- 不存在的文件返回 404。
- 非 GET 请求返回 501。
- 畸形请求返回 400。
- 目录遍历攻击不能逃逸文档根目录。
- 单元测试和冒烟测试通过。

## 阶段二：非阻塞 I/O 与连接状态优化

目标：提升连接处理的健壮性，确保大文件、慢客户端和并发请求场景下行为稳定。

主要开发任务：

- 完善连接状态机，将连接处理拆分为更明确的阶段。
- 建议状态包括 `READING`、`PROCESSING`、`WRITING_HEADER`、`WRITING_FILE`、`CLOSING`。
- 更系统地处理 partial read。
- 更系统地处理 partial write。
- 将响应头发送和文件内容发送拆成更清晰的流程。
- 根据连接状态动态切换 epoll 监听事件，例如 `EPOLLIN` 和 `EPOLLOUT`。
- accept 循环处理到 `EAGAIN`。
- read 循环处理到 `EAGAIN`。
- write 循环处理到 `EAGAIN`。
- 增加最大请求头大小限制。
- 增加最大连接数限制。
- 增加读写错误处理和资源释放审计。
- 增加空闲连接超时，避免恶意连接长期占用 fd。
- 增加更多异常场景测试，例如客户端中途断开、请求头过长、慢请求、大文件下载。

验收标准：

- 大文件下载不会截断。
- 慢客户端不会阻塞主事件循环。
- 高并发请求下 fd 能正确释放。
- 错误请求不会导致服务器崩溃。
- 冒烟测试和基础压测稳定通过。

## 阶段三：高性能优化版

目标：在业务逻辑稳定的基础上提升吞吐量、降低 CPU 消耗，并形成可量化的压测指标。

主要开发任务：

- 使用 `sendfile` 发送静态文件，减少用户态和内核态之间的数据拷贝。
- 对比 `read/write` 和 `sendfile` 的性能差异。
- 根据压测结果评估是否启用 `EPOLLET` 边缘触发。
- 如果启用 `EPOLLET`，确保所有 accept、read、write 都严格循环到 `EAGAIN`。
- 优化响应头构造，减少重复格式化和内存分配。
- 优化缓冲区复用策略，减少频繁申请和释放内存。
- 优化 MIME 类型查找。
- 增加更多可配置项，例如 `BACKLOG`、`MAX_EVENTS`、`MAX_CONNECTIONS`、`REQUEST_TIMEOUT`。
- 完善压测脚本，优先支持 `wrk`，兼容 `ab`。
- 记录核心性能指标，包括 QPS、平均延迟、P95/P99 延迟、错误率、CPU 占用和内存占用。
- 使用压测结果指导下一轮优化，而不是盲目修改代码。

验收标准：

- 小文件高并发吞吐量较阶段二明显提升。
- 中等文件下载时 CPU 占用下降。
- 高并发压测下错误率可控。
- 服务器可以持续运行，不出现明显 fd 泄漏或内存增长。
- 压测脚本可以稳定复现测试结果。

## 阶段四：工程完善与扩展准备

目标：提升项目可维护性、可学习性和可扩展性，为后续功能开发打好基础。

主要开发任务：

- 完善架构文档，详细说明 Reactor 模型、模块职责、请求处理流程和连接状态机。
- 完善 README，使新手可以独立完成编译、运行、测试和源码阅读。
- 增加更完整的异常测试。
- 增加访问日志，记录请求方法、URI、状态码、响应大小和处理耗时。
- 梳理公共接口，减少模块之间不必要的耦合。
- 为 keep-alive 预留连接复用能力。
- 为目录索引功能预留静态文件处理接口。
- 为线程池或多 Reactor 模型预留演进空间。
- 为 TLS 或反向代理等高级功能保留清晰边界。
- 整理代码风格，保证命名、错误处理和资源释放方式一致。

验收标准：

- 新开发者可以通过文档理解项目整体结构。
- 测试覆盖常见正常路径和异常路径。
- 后续新增功能时不需要大规模重构现有模块。
- 项目具备继续演进为高性能静态服务器的基础。

## 推荐开发顺序

1. 先完成阶段一，保证服务器完整可运行。
2. 再进入阶段二，补强连接状态和非阻塞 I/O 细节。
3. 等业务逻辑稳定后进入阶段三，通过压测数据指导性能优化。
4. 最后进入阶段四，完善文档、测试、日志和扩展边界。
